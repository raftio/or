# RFC-002: Execution Bundle Schema

**Status:** Draft  
**Summary:** Defines the machine-readable schema for the execution bundle: structure (spec → tasks, dependencies), versioning, and mapping from product spec.

---

## 1. Overview

The execution bundle is the primary artifact produced by the Control Plane and consumed by the Execution Plane (dev/IDE/agent). This RFC defines its schema only; the engine that produces bundles is defined in RFC-014.

**In scope:**

- Structure of the bundle (spec reference, tasks, dependencies, acceptance criteria references).
- Versioning and stable identity.
- How the bundle maps from product spec sources (Confluence/Notion + Jira/Linear).

**Out of scope:**

- API to fetch or submit bundles (RFC-006).
- Bundling engine logic and when to re-bundle (RFC-014).

---

## 2. Schema

### 2.1 Structure

A bundle is a single document (e.g. JSON or YAML) containing:

| Field | Description |
|-------|-------------|
| `id` | Stable unique identifier (e.g. UUID or `{ticket_id}:{version}`). |
| `version` | Monotonic version for the same logical bundle (e.g. integer or semantic). |
| `spec_ref` | Reference to the source spec (document URL or provider-specific id from Confluence/Notion). |
| `ticket_ref` | Reference to the ticket (e.g. Jira key, Linear id). |
| `tasks` | Ordered list of work items derived from the spec; each has an id, title, optional description. |
| `dependencies` | Optional graph or list of task dependencies (e.g. task B depends on task A). |
| `acceptance_criteria_refs` | References to acceptance criteria (ids from RFC-003) that this bundle is expected to satisfy. |
| `context` | Optional synthesized context (excerpts from docs, related tickets) used by the engine (RFC-008). |
| `created_at` / `updated_at` | Timestamps. |

Extensions (e.g. repos in scope for multi-repo) can be added in a reserved `ext` or `meta` field.

### 2.2 Versioning and identity

- `id` uniquely identifies the bundle within the Control Plane.
- `version` increases when the bundle is regenerated for the same ticket/spec (e.g. spec or ticket updated). Consumers can cache by `(id, version)` and invalidate when version changes.
- Idempotency: producing a bundle with the same inputs (spec + ticket + context snapshot) should yield the same `(id, version)` or a predictable new version.

### 2.3 Mapping from product spec

- **Spec (Confluence/Notion):** `spec_ref` points to the source document. Content is fetched via RFC-010; the bundling engine (RFC-014) maps sections/headings to tasks and extracts or links acceptance criteria.
- **Ticket (Jira/Linear):** `ticket_ref` links the bundle to the ticket. Title, description, and ticket-level acceptance criteria are available via RFC-009 and feed into `tasks` and `acceptance_criteria_refs`.
- Mapping rules (e.g. “one heading = one task”) are policy/engine concern (RFC-014); this RFC only defines the target shape of the bundle.
