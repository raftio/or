# RFC-003: Acceptance Criteria Model

**Status:** Draft  
**Summary:** Defines the model for acceptance criteria: structure (ID, description, verification conditions), link to requirement, and format for programmatic verification.

---

## 1. Overview

Acceptance criteria (AC) are the contract between product intent and implementation. This RFC defines their structure and how they link to requirements and to programmatic verification (tests, assertions). Traceability from AC to code and tests is detailed in RFC-015.

**In scope:**

- Structure of a single acceptance criterion (id, description, verification condition type).
- Link to requirement (ticket and/or spec).
- Format for programmatic verification (assertions, test references).

**Out of scope:**

- How AC are stored or queried at scale (RFC-015).
- How evidence is linked to AC (RFC-004, RFC-017).

---

## 2. Model

### 2.1 Structure

Each acceptance criterion has:

| Field | Description |
|-------|-------------|
| `id` | Stable unique id (e.g. within a ticket or bundle: `{ticket_id}/ac/{local_id}`). |
| `description` | Human-readable description (one or more sentences). |
| `verification_condition` | Optional structured condition for programmatic check (see 2.3). |
| `requirement_ref` | Link to requirement: ticket id and/or spec fragment (e.g. section id, heading). |
| `source` | Origin: e.g. `ticket`, `spec`, `derived` (from bundling). |

### 2.2 Link to requirement

- **Ticket:** `requirement_ref` includes the ticket key/id (Jira/Linear). AC may be authored in the ticket or imported from a spec.
- **Spec:** `requirement_ref` may include document id and fragment (e.g. Confluence page + block, or Notion block id) so that traceability UI can jump to the source.
- One requirement (ticket or spec section) can have multiple AC; one AC is associated with one primary requirement.

### 2.3 Format for programmatic verification

To support “evidence validates AC” (RFC-016, RFC-017), an AC can carry:

- **Assertion type:** e.g. `test_pass`, `coverage_threshold`, `manual`. 
- **Test reference:** Optional pointer to test(s) that satisfy this AC (e.g. test suite name, test id, or tag). Format is provider-agnostic; adapters (RFC-012) map CI/test output to these refs.
- **Parameters:** e.g. minimum coverage percentage, or key-value expectations.

If `verification_condition` is absent, the AC is considered manually verified or not yet machine-verifiable.
