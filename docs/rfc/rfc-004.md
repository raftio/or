# RFC-004: Evidence Payload Format

**Status:** Draft  
**Summary:** Defines the unified format for evidence: test results, coverage reports, CI logs, validation signals; schema and lifecycle (created → validated → linked).

---

## 1. Overview

Evidence is produced by the Execution Plane (CI, tests, coverage) and consumed by the Control Plane for validation and traceability. This RFC defines a single payload format so that multiple sources (different repos, CI systems) can report in a consistent way. The Evidence Sync Engine (RFC-017) collects and stores these payloads; multi-repo aggregation uses the same format with required linking fields.

**In scope:**

- Schema of an evidence payload (test results, coverage, CI logs, validation signals).
- Lifecycle states: created → validated → linked.
- Linking to PR, commit, ticket, and bundle.

**Out of scope:**

- Transport (push vs pull) and API (RFC-012, RFC-006).
- Storage, dedup, and aggregation logic (RFC-017).

---

## 2. Schema

### 2.1 Payload structure

A single evidence payload contains:

| Field | Description |
|-------|-------------|
| `id` | Optional client-generated id for idempotency (e.g. CI job id). |
| `repo` | Repository identifier (e.g. `org/repo` or provider-specific id). Required for multi-repo. |
| `branch` / `commit_sha` | Branch and/or commit this evidence applies to. |
| `pr_id` | Optional pull request id (provider-specific) when evidence is for a PR build. |
| `ticket_id` | Ticket key/id (e.g. Jira key). Required so OR can aggregate by ticket across repos. |
| `test_results` | Summary: pass/fail/skip counts; optional list of failed test names or ids. |
| `coverage` | Optional coverage summary (e.g. line %, branch %); optional per-file. |
| `ci_logs` | Optional reference (URL or blob id) to CI run logs. |
| `validation_signals` | Optional list of signals (e.g. lint passed, security scan passed). |
| `ci_status` | Overall status: e.g. `success`, `failure`, `cancelled`. |
| `artifact_urls` | Optional URLs to artifacts (e.g. coverage report, test report). |
| `timestamp` | When the evidence was produced (e.g. CI job end time). |

**Multi-repo:** For microservices or multi-repo setups, every payload must include `repo`, `ticket_id`, and either `commit_sha` or `pr_id` so the Control Plane can aggregate evidence per ticket and per repo.

### 2.2 Lifecycle

- **created:** Payload received and stored; not yet evaluated against acceptance criteria or bundle.
- **validated:** Control Plane has evaluated (e.g. pass/fail, coverage threshold, AC coverage); result may be stored on the payload or in a separate validation result.
- **linked:** Evidence is linked to a specific PR, bundle, and/or acceptance criteria for traceability and PR Intelligence (RFC-016).

Transitions are defined by the state machine (RFC-007) and Evidence Sync Engine (RFC-017); this RFC only defines the lifecycle labels.

### 2.3 Linking

- **To PR:** `pr_id` + `repo` (and optionally provider) identify the PR. Reverse link: PR → list of evidence payloads.
- **To commit:** `repo` + `commit_sha` identify the commit.
- **To ticket:** `ticket_id` links to the ticket; used for per-ticket evidence view and “evidence complete” rules (RFC-017).
- **To bundle:** Optional `bundle_id` / `bundle_version` when the run was for a known bundle.
