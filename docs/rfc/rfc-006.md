# RFC-006: Control Plane API

**Status:** Draft  
**Summary:** Defines the Control Plane API surface: context synthesis, execution planning, evidence validation, workflow orchestration, outcome tracking; auth and versioning.

---

## 1. Overview

The Control Plane exposes capabilities to integrations (IDE, CI, ticket/docs providers) and to Orca’s own UI. This RFC defines the API surface by capability area and principles for auth and versioning; concrete HTTP/REST or GraphQL design can be a later refinement.

**In scope:**

- Capability areas and their scope (what the API allows).
- Authentication and authorization principles.
- Versioning and compatibility principles.

**Out of scope:**

- Detailed endpoint specs or request/response schemas (can follow in an API spec doc).
- Identity provider integration (RFC-020).

---

## 2. Capability areas

### 2.1 Context synthesis

- **Input:** Ticket id(s), optional spec/document refs.
- **Output:** Synthesized context (text, links) used by the bundling engine and by Idea/Spec Assist. May be exposed as “get context for ticket” or as part of bundle creation.
- **Related:** RFC-008 (protocol), RFC-009, RFC-010, RFC-011.

### 2.2 Execution planning

- **Input:** Ticket id (and optionally spec ref).  
- **Output:** Execution bundle (RFC-002) or a reference to it. Includes “create/update bundle” and “get bundle for ticket”.
- **Related:** RFC-014 (engine), RFC-002.

### 2.3 Evidence validation

- **Input:** Evidence payload(s) (RFC-004) or reference to stored evidence; optional PR/ticket id.
- **Output:** Validation result (e.g. pass/fail, AC coverage, risk flags). Used by PR Intelligence (RFC-016).
- **Related:** RFC-004, RFC-017, RFC-016.

### 2.4 Workflow orchestration

- **Input:** Triggers (e.g. ticket updated, PR opened); optional manual actions (e.g. “request review”).
- **Output:** Status updates, notifications, or state transitions. Supports handoff and latency reduction (RFC-018).
- **Related:** RFC-007 (state machine), RFC-018.

### 2.5 Outcome tracking

- **Input:** Release id, optional KPI/outcome definitions (RFC-005).
- **Output:** Outcome records, post-release insights, or attribution (ticket → outcome).
- **Related:** RFC-019, RFC-005.

---

## 2.6 Auth and versioning principles

- **Authentication:** All API access is authenticated (e.g. API key, OAuth, or platform identity). RFC-020 defines tenancy and isolation; API must enforce tenant context.
- **Authorization:** Actions are scoped by tenant and by role/capability (e.g. read bundles vs write evidence). Fine-grained permissions (e.g. per-project) are defined in RFC-020.
- **Versioning:** API has a version prefix or header (e.g. `/v1/...` or `Accept-Version: 1`). Breaking changes require a new version; additive changes may be backward-compatible within the same version.
