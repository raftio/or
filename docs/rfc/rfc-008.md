# RFC-008: Context Synthesis Protocol

**Status:** Draft  
**Summary:** Defines how to synthesize context from multiple sources (tickets, docs, specs, PR metadata); precedence, merging, and cache/invalidation.

---

## 1. Overview

The Control Plane needs a single, coherent context for a ticket (and optionally spec) to produce execution bundles and to support Idea/Spec Assist. Context is pulled from ticket systems, document systems, and Git (PR metadata). This RFC defines the protocol: sources, precedence, merging, and cache/invalidation. Actual adapters are defined in RFC-009, RFC-010, RFC-011.

**In scope:**

- Sources of context and what each provides.
- Precedence and merging rules when the same information appears in multiple places.
- Cache and invalidation so that context stays fresh without over-fetching.

**Out of scope:**

- Adapter implementation details (RFC-009–011).
- Bundling engine logic that consumes context (RFC-014).

---

## 2. Protocol

### 2.1 Sources

| Source | Provides | Adapter |
|--------|----------|---------|
| **Tickets (Jira/Linear)** | Title, description, acceptance criteria, status, links to docs/PRs. | RFC-009 |
| **Documents (Confluence/Notion)** | Spec body, sections, embedded AC, links. | RFC-010 |
| **Git / PR metadata** | PR title, description, linked ticket refs, branch name, commit messages. | RFC-011 |

Context synthesis is triggered when building a bundle (RFC-014) or when serving “context for ticket” (e.g. for Idea/Spec Assist). Input: ticket id and optional spec/document ref.

### 2.2 Precedence and merging rules

- **Ticket vs spec:** If both ticket and spec describe acceptance criteria, define a rule: e.g. “spec overrides ticket for AC” or “merge both and deduplicate by text similarity.” Same for title/description: e.g. ticket title + spec body, or spec title if present.
- **Multiple docs:** If a ticket links to several docs, all are fetched; order can be by link order or by “primary” link. Conflicting sections (e.g. two “Scope” sections) use first or explicit precedence (e.g. Confluence over Notion).
- **PR metadata:** When a PR exists, PR description and title can supplement or override ticket title/description for “what this PR does” in the context; ticket remains the authority for AC and status.
- **Explicit rules:** Document the chosen precedence in the protocol so that all consumers (bundling engine, UI) behave consistently.

### 2.3 Cache and invalidation

- **Cache key:** e.g. `(ticket_id, spec_ref, optional PR id)` plus adapter-specific resource versions (e.g. ticket updated_at, document last_edited).
- **TTL:** Optional short TTL (e.g. minutes) for context used in UI; longer or no TTL when context is pinned at bundle creation time (bundle stores a snapshot).
- **Invalidation:** On webhook or poll: ticket_updated, document_updated, pr_updated. Invalidate cache for affected (ticket_id, …) so the next synthesis fetches fresh data.
- **Snapshot in bundle:** Once a bundle is created, the context used for that bundle is stored as part of the bundle (or by reference) and is not invalidated; only new bundles get new context.
