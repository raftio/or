# RFC-009: Ticket Provider Adapter (Jira / Linear)

**Status:** Draft  
**Summary:** Defines the contract and adapter for ticket systems: read tickets/specs, write status, link execution bundles and evidence; webhook/polling.

---

## 1. Overview

The Control Plane needs to read ticket data (title, description, acceptance criteria, status) and optionally write back status and links (e.g. link to execution bundle or evidence). This RFC defines the adapter contract for Jira and Linear (and similar systems). Context Synthesis (RFC-008) and the Bundling Engine (RFC-014) consume ticket data; the adapter hides provider-specific APIs.

**In scope:**

- Read: tickets, specs (if stored in ticket), and links.
- Write: status updates, link execution bundle/evidence to ticket.
- Webhook vs polling for sync and invalidation.

**Out of scope:**

- Document providers (Confluence/Notion) — RFC-010.
- Git/PR linking details — RFC-011; adapter may expose “PRs linked to ticket” if provider supports it or via Git adapter.

---

## 2. Contract

### 2.1 Read

- **Get ticket:** By ticket key (Jira) or id (Linear). Returns: id/key, title, description, status, acceptance criteria (if in ticket body or custom field), links to docs, assignee, timestamps.
- **List tickets:** Optional: by project, filter, or query (e.g. “ready for dev”). Used for dashboards or bulk context.
- **Specs:** If the ticket system stores spec content or rich description, that is returned as part of ticket; otherwise spec comes from document provider (RFC-010) via link in ticket.

Adapter normalizes to a common in-memory or DTO shape so that RFC-008 and RFC-014 do not depend on Jira/Linear specifics.

### 2.2 Write

- **Update status:** Set ticket status (e.g. In Progress, In Review, Done) when state machine transitions (RFC-007).
- **Link execution bundle:** Store a link or custom field pointing to the bundle (e.g. URL to Orca bundle view or bundle id).
- **Link evidence:** Optional: attach or link evidence summary (e.g. “Evidence: 3/3 passed”) or URL to evidence view.

Write operations are best-effort; failures should not block Control Plane flow (e.g. log and retry or queue).

### 2.3 Webhook vs polling

- **Webhooks:** When supported (e.g. Jira webhooks, Linear subscriptions), subscribe to ticket created/updated/deleted. Use for cache invalidation (RFC-008) and state machine events (RFC-007). Payload should include ticket id and optionally changed fields.
- **Polling:** When webhooks are unavailable or unreliable, poll periodically (e.g. by project or by “recently updated”). Same use: invalidation and events.
- **Recommendation:** Prefer webhooks for low latency; fall back to polling with a reasonable interval to avoid rate limits.
