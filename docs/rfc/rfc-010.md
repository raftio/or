# RFC-010: Document Provider Adapter (Confluence / Notion)

**Status:** Draft  
**Summary:** Defines the contract for document sources: fetch specs/docs, mapping spec → execution bundle input, sync and permissions.

---

## 1. Overview

Specifications and product docs often live in Confluence or Notion. The Control Plane needs to fetch these documents to build execution bundles (RFC-014) and to support Idea/Spec Assist. This RFC defines the adapter contract: what to fetch, how it maps to bundle input, and sync/permissions considerations.

**In scope:**

- Fetch specs/docs by reference (page id, URL, or link from ticket).
- Mapping spec content to execution bundle input (structure usable by RFC-002, RFC-014).
- Sync and permissions (read access, rate limits).

**Out of scope:**

- Ticket system (RFC-009); document links may come from tickets.
- Bundling engine logic (RFC-014) — only the shape of “spec as input” is defined here.

---

## 2. Contract

### 2.1 Fetch specs/docs

- **By reference:** Input is a document reference: Confluence (space key + page id or URL), Notion (page id or URL). Adapter resolves to raw or structured content.
- **Content format:** Return structured content when possible (e.g. headings, sections, lists, blocks) so that the bundling engine can map sections to tasks and extract AC. Fallback: plain text or HTML for simple parsing.
- **Links:** Resolve or expose in-document links (e.g. to other pages, to Jira tickets) so that Context Synthesis (RFC-008) can follow them if needed.
- **Metadata:** Page title, last modified, author (for cache invalidation and display).

### 2.2 Mapping spec → execution bundle input

- **Sections → tasks:** Document structure (headings, numbered sections) suggests task boundaries. Adapter or a thin mapping layer outputs a list of “sections” with id, title, and body (or ref) so that RFC-014 can produce `tasks` (RFC-002) and optional `acceptance_criteria_refs`.
- **AC extraction:** If the spec contains explicit acceptance criteria (e.g. “AC: …” or a dedicated section), expose them as structured items (id, description) compatible with RFC-003 so they can be linked into the bundle.
- **Spec ref in bundle:** The execution bundle’s `spec_ref` (RFC-002) must uniquely identify this document (e.g. Confluence page id, Notion page id) for traceability and re-fetch.

### 2.3 Sync and permissions

- **Authentication:** Adapter uses tenant-configured credentials (e.g. API token, OAuth) with read-only access where possible.
- **Permissions:** Only fetch documents the identity can read; permission errors are surfaced so that Orca can show “no access” instead of empty content.
- **Rate limits:** Respect provider rate limits (e.g. Confluence/Notion API). Use cache (RFC-008) to avoid repeated fetches; invalidate on document_updated webhook or poll.
- **Sync trigger:** Fetch on demand when building a bundle or serving context; optional background sync for “recent” or “linked from tickets” docs.
