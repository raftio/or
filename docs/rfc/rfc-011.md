# RFC-011: Git Provider Adapter (GitHub / GitLab)

**Status:** Draft  
**Summary:** Defines the contract for Git: PR metadata, code diffs, branch/tag, link PR ↔ ticket ↔ evidence; webhooks and API usage.

---

## 1. Overview

The Control Plane needs to know about PRs, branches, commits, and to link them to tickets and evidence. This RFC defines the adapter contract for GitHub and GitLab (and similar). Evidence payloads (RFC-004) and PR Intelligence (RFC-016) depend on PR and repo identity; Context Synthesis (RFC-008) may use PR description/title.

**In scope:**

- PR metadata (title, description, state, branch, base, head).
- Code diffs (optional; for traceability and risk).
- Branch/tag and commit resolution.
- Link PR ↔ ticket ↔ evidence (how links are established and stored).
- Webhooks and API usage (rate limits, events).

**Out of scope:**

- CI/test integration (RFC-012) — Git adapter does not run CI; it provides PR/commit identity for evidence linking.
- IDE integration (RFC-013) — IDE may use Control Plane API that internally uses this adapter.

---

## 2. Contract

### 2.1 PR metadata, code diffs, branch/tag

- **PR metadata:** For a given PR (repo + pr number or id): title, description, state (open/merged/closed), source branch, target branch, head commit sha, author, timestamps. Normalize across GitHub/GitLab.
- **Code diffs:** Optional: fetch diff patch or file list for the PR. Used for traceability (RFC-015) and risk/scope in PR Intelligence (RFC-016). Large diffs may be summarized or fetched on demand.
- **Branch/tag:** Resolve branch name or tag to commit sha; list branches/tags for a repo if needed (e.g. “repos in scope” for a ticket).
- **Commit:** Get commit metadata (sha, message, author, timestamp) by repo + sha.

### 2.2 Link PR ↔ ticket ↔ evidence

- **PR → ticket:** Extract ticket id from PR title, description, or branch name (e.g. `PROJ-123`, `feature/PROJ-123-add-x`). Store as (repo, pr_id, ticket_id). Ticket adapter (RFC-009) may also expose “linked PRs” if the provider supports it; Git adapter is the source of truth for PR-side link.
- **PR ↔ evidence:** Evidence payloads (RFC-004) carry `repo`, `pr_id`, `ticket_id`. Control Plane joins evidence to PR by (repo, pr_id). Reverse: given a PR, return all evidence for that PR (from Evidence Sync, RFC-017).
- **Persistence:** Links are stored in the Control Plane (not necessarily in the Git provider). Adapter provides read-only PR/ticket association; write-back to Git (e.g. comment with bundle link) is optional and can be a separate capability.

### 2.3 Webhooks and API usage

- **Webhooks:** Subscribe to PR opened, updated, merged, closed; push to branch; optionally release/tag. Use for state machine (RFC-007) and cache invalidation. Payload must include repo, PR id, and enough data to resolve ticket id.
- **API:** Use provider REST/GraphQL for fetch-by-id and diff. Respect rate limits; use conditional requests or ETag where supported to reduce calls.
- **Multi-repo:** Adapter is per-repo or per-org; Orqestra aggregates across repos using a single tenant config (list of repos to watch).
