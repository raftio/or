# RFC-012: CI/CD & Test Integration

**Status:** Draft  
**Summary:** Defines how to receive test results, coverage, CI logs from pipelines; push vs pull, payload format (RFC-004), and linking to PR/bundle.

---

## 1. Overview

Evidence is produced by CI/CD and test runs. This RFC defines how that evidence reaches the Control Plane: push (CI calls Orqestra) vs pull (Orqestra polls CI), the payload format (RFC-004), and how payloads are linked to PR/bundle/ticket. The Evidence Sync Engine (RFC-017) consumes these inputs.

**In scope:**

- Push vs pull models and when to use each.
- Payload format (reference RFC-004) and any CI-specific extensions.
- Linking evidence to PR, bundle, and ticket (how CI discovers ticket/PR context).

**Out of scope:**

- Storage, dedup, and aggregation (RFC-017).
- Validation logic (RFC-016, RFC-017).

---

## 2. Integration

### 2.1 Push vs pull

- **Push (recommended):** CI pipeline, at the end of a job (build + test + coverage), sends an evidence payload to the Control Plane (HTTP POST to a well-known endpoint). Requires CI to have Orqestra endpoint and credentials. Advantages: real-time, no polling.
- **Pull:** Control Plane periodically polls CI provider APIs (e.g. GitHub Actions runs, GitLab CI jobs) and maps job results to RFC-004 payloads. Use when push is not possible (e.g. restricted CI environment). Must respect rate limits and handle pagination.
- **Hybrid:** Push for primary pipeline; pull for backup or for providers without webhook/callback support.

### 2.2 Payload format (RFC-004)

- Every payload must conform to RFC-004 (repo, branch/commit_sha, pr_id, ticket_id, test_results, coverage, ci_status, timestamp, etc.).
- CI adapter or script in the pipeline is responsible for building the payload: extract ticket id from branch name or PR; get test/coverage from job artifacts or API; set ci_status from job result.
- Optional: provider-specific fields in an `ext` or `meta` block (e.g. job url, workflow name) for debugging and display.

### 2.3 Linking to PR/bundle/ticket

- **Ticket id:** CI must determine ticket id (e.g. from branch name `feature/PROJ-123`, or from PR description/title). Passed in payload as `ticket_id`.
- **PR id:** If the run is for a PR, set `pr_id` (and repo) so that evidence is attached to the PR (RFC-017). Control Plane can also infer PR from branch if the provider API supports it (e.g. “branch X is head of PR Y”).
- **Bundle:** Optional. If the run was triggered with a known bundle id/version (e.g. IDE or orchestrator passed it), include in payload for traceability; otherwise Control Plane may link later by (ticket_id, repo, commit).
