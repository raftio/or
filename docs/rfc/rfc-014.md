# RFC-014: Context Bundling Engine

**Status:** Draft  
**Summary:** Details the engine that converts spec + tickets into execution bundle: input (spec + tickets), output (RFC-002), rules, idempotency, and when to re-bundle.

---

## 1. Overview

The Context Bundling Engine is the Control Plane component that produces execution bundles (RFC-002) from product specs and tickets. It consumes synthesized context (RFC-008) and ticket/document data (RFC-009, RFC-010) and outputs a single, versioned bundle for the Execution Plane.

**In scope:**

- Inputs: spec (from document provider), ticket(s), and optional context snapshot.
- Output: Execution bundle conforming to RFC-002.
- Rules for mapping (e.g. sections → tasks, AC extraction) and when to re-bundle; idempotency.

**Out of scope:**

- Schema of the bundle (RFC-002).
- Context synthesis protocol (RFC-008).
- API to trigger or fetch bundle (RFC-006).

---

## 2. Engine

### 2.1 Input

- **Spec:** Fetched via RFC-010 (Confluence/Notion). May be identified by ticket link or explicit ref. Content: sections, headings, body, embedded AC.
- **Tickets:** Fetched via RFC-009 (Jira/Linear). At least one ticket id; returns title, description, AC, status.
- **Context snapshot:** Optional pre-synthesized context from RFC-008 (merged excerpts, related tickets) to avoid re-fetch inside the engine.

Engine may trigger context synthesis (RFC-008) internally when building a bundle, or receive context from a prior step.

### 2.2 Output

- **Bundle:** Conforms to RFC-002: id, version, spec_ref, ticket_ref, tasks, dependencies, acceptance_criteria_refs, context (optional), timestamps.
- **Tasks:** Derived from spec sections and/or ticket breakdown. Each task has id, title, optional description. Order preserved.
- **Acceptance criteria refs:** Point to AC ids (RFC-003) that this bundle is expected to satisfy; AC may come from spec or ticket.
- **Versioning:** New bundle version when input has changed (spec or ticket updated) or when re-bundle is explicitly requested. Same input → same output for idempotency (deterministic version or no-op).

### 2.3 Rules and when to re-bundle

- **Mapping rules:** Configurable or default: e.g. “one H2 section = one task”, “AC section in spec → list of AC refs”, “ticket AC override or merge with spec AC”. Rules are engine policy; this RFC only requires that the output is valid per RFC-002.
- **When to re-bundle:** (1) Ticket created or updated (webhook/poll from RFC-009). (2) Spec document updated (RFC-010). (3) Manual “regenerate bundle” from UI or API. (4) Optional: periodic re-bundle for “in progress” tickets to pick up spec edits.
- **Idempotency:** For the same (ticket_id, spec_ref, context version), producing a bundle twice should yield the same bundle id and version (or one no-op). Use deterministic id/version (e.g. hash of inputs + schema version) or compare previous bundle and skip if inputs unchanged.
