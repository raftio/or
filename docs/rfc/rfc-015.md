# RFC-015: Acceptance Criteria Traceability

**Status:** Draft  
**Summary:** Defines how to map acceptance criteria (RFC-003) to code diffs and test coverage; storage model, query API, and UI contract for traceability view.

---

## 1. Overview

Traceability connects acceptance criteria to implementation (code diffs) and verification (test coverage, evidence). This RFC defines the storage model, how to record and query these links, and the UI contract for a traceability view. PR Intelligence (RFC-016) and Evidence Sync (RFC-017) produce and consume traceability data.

**In scope:**

- Storage model: how AC ↔ code and AC ↔ tests/evidence are stored.
- Query API: get traceability for a ticket, PR, or AC.
- UI contract: what data the traceability view needs (so frontend and backend stay aligned).

**Out of scope:**

- AC model itself (RFC-003).
- Evidence payload format (RFC-004).
- How code diffs are fetched (RFC-011); this RFC defines how links are stored and queried.

---

## 2. Traceability

### 2.1 Storage model

- **AC ↔ code:** For each acceptance criterion, store zero or more “implementations”: (repo, commit_sha or pr_id, file path, optional range or hunk). Source: diff analysis (e.g. from PR diff) or manual/assisted tagging. Supports multiple repos (multi-repo: same AC can be implemented in more than one repo).
- **AC ↔ tests / evidence:** Store (ac_id, evidence_id or test_ref). Evidence payloads (RFC-004) may reference tests; when evidence is linked to AC (RFC-017), the traceability record is created. Optional: map test name/id to AC via tags or config (e.g. “test tagged PROJ-123-AC1”).
- **Ticket/PR aggregate:** Traceability view often needs “all AC for ticket X” and “for each AC: code refs + evidence refs”. Storage can be normalized (AC table, implementation_links table, evidence_links table) or denormalized for read; query API (2.2) defines the shape.

### 2.2 Query API

- **By ticket:** Return all AC for the ticket (from bundle or ticket), and for each AC: list of code refs (repo, pr, file, range), list of evidence/test refs, and status (e.g. “linked”, “verified”).
- **By PR:** Return all AC that have implementation or evidence in this PR (repo + pr_id). Used by PR Intelligence (RFC-016).
- **By AC:** Return all code refs and evidence refs for a single AC id.
- **Filters:** Optional filters (e.g. only verified, only from CI evidence) for UI.

### 2.3 UI contract (traceability view)

- **Input:** Ticket id (or PR id for “traceability for this PR”).
- **Output:** Tree or list: AC → code locations (clickable links to repo/PR/file) and AC → tests/evidence (status, link to evidence view). Optional: coverage indicator (e.g. “3/5 AC verified”).
- **Refresh:** When new evidence is linked or new PR is merged, traceability view should reflect updates (via re-query or subscription). UI does not compute links; it consumes the query API output.
