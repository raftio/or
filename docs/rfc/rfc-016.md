# RFC-016: PR Intelligence Layer

**Status:** Draft  
**Summary:** Defines PR summaries, risk flags, evidence validation before review: input (PR + evidence + criteria), output format, and integration with review workflow. Includes per-PR and per-ticket view for multi-repo.

---

## 1. Overview

Before a human reviews a PR, OR can provide a summary, risk flags, and evidence validation status. This RFC defines the inputs (PR, evidence, acceptance criteria), the output format, and how it integrates with the review workflow. In multi-repo setups, both per-PR and per-ticket (aggregate) views are supported.

**In scope:**

- Input: PR metadata, linked evidence, and acceptance criteria (from bundle or ticket).
- Output: summary text, risk flags (e.g. large diff, no tests, AC not covered), evidence validation status (pass/fail, coverage).
- Integration: how reviewers see this (e.g. comment, status check, OR UI). Per-PR and per-ticket view for multi-repo.

**Out of scope:**

- Evidence collection (RFC-017).
- Traceability storage (RFC-015) — PR Intelligence consumes traceability and evidence.

---

## 2. Layer

### 2.1 Input

- **PR:** Repo + pr_id (and ticket_id from link). Metadata from Git adapter (RFC-011): title, description, diff summary (files changed, lines), author.
- **Evidence:** All evidence payloads linked to this PR (from RFC-017): test results, coverage, CI status. For per-ticket view: evidence from all PRs linked to the ticket (multiple repos).
- **Criteria:** Acceptance criteria for the ticket (or bundle): list of AC ids and descriptions (RFC-003). Used to compute “AC covered” and to show what’s missing.

### 2.2 Output format

- **Summary:** Short prose or bullet summary of “what this PR does” (e.g. derived from title, description, or diff). Optional: list of tasks (from bundle) touched.
- **Risk flags:** List of flags, e.g. `large_diff`, `no_tests`, `ac_not_covered`, `ci_failed`, `coverage_below_threshold`. Each flag may have severity and optional detail (e.g. “3/5 AC have no linked evidence”).
- **Evidence validation:** Overall status: e.g. `pass`, `fail`, `partial`. Per-criterion or per-repo breakdown: which AC have evidence, which passed. For multi-repo: “Ticket PROJ-123: 3/3 repos have evidence, 2 pass 1 fail” plus link to each PR’s evidence.
- **Structured:** Output is machine-readable (JSON) for status checks and UI; optional human-readable snippet for PR comment.

### 2.3 Integration with review workflow

- **Per-PR:** Reviewer opens PR in Git provider; sees OR summary and status via (1) PR comment posted by OR, (2) status check (e.g. “OR: evidence pass”), or (3) link to OR UI for this PR. Reviewer uses it to decide review focus or to approve.
- **Per-ticket (multi-repo):** In OR UI, “Ticket PROJ-123” view shows all PRs for the ticket and aggregate evidence/risk. Reviewer can open each PR or see “all evidence complete and pass” for the ticket.
- **Trigger:** PR Intelligence runs when (1) evidence is received/updated for the PR (RFC-017), or (2) PR is opened/updated. Result is stored and exposed via API (RFC-006) and/or pushed to Git (comment, status).
