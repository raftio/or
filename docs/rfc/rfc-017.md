# RFC-017: Evidence Sync Engine

**Status:** Draft  
**Summary:** Defines how to collect and sync evidence from tests, coverage, CI into a unified view; sources (RFC-012), storage, dedup, link to PR/bundle/ticket. Includes multi-repo: aggregation key (ticket_id + repo + PR), unified view per ticket, and “evidence complete” rules.

---

## 1. Overview

The Evidence Sync Engine collects evidence payloads (RFC-004) from CI and other sources (RFC-012), stores them, deduplicates, and links them to PRs, bundles, and tickets. It exposes a unified view for validation (RFC-016) and traceability (RFC-015). For microservices or multi-repo setups, aggregation by ticket and “evidence complete” rules are defined here.

**In scope:**

- Sources: how evidence is received (RFC-012); storage and dedup.
- Linking to PR, bundle, ticket.
- Multi-repo: aggregation key (ticket_id + repo + PR), unified view per ticket, and when to consider “evidence complete” for a ticket.

**Out of scope:**

- Evidence payload schema (RFC-004).
- Validation logic (RFC-016) — Engine only stores and links; validation consumes the unified view.

---

## 2. Engine

### 2.1 Sources (RFC-012)

- Evidence arrives via push (CI POST) or pull (Control Plane polls CI). Each payload conforms to RFC-004.
- Engine receives payloads via Control Plane API (RFC-006) or internal queue. Validates required fields (repo, ticket_id, commit_sha or pr_id, ci_status, timestamp).

### 2.2 Storage and dedup

- **Storage:** Persist each payload (or normalized record) with id, ticket_id, repo, pr_id, commit_sha, timestamps, and payload fields. Index by ticket_id, (repo, pr_id), (repo, commit_sha) for fast lookup.
- **Dedup:** Same (repo, commit_sha, job_id or idempotency key) → update existing record (e.g. re-run of CI) rather than duplicate. Use upsert semantics.

### 2.3 Link to PR / bundle / ticket

- **Ticket:** Every payload has ticket_id; all evidence for a ticket is queryable by ticket_id (unified view per ticket).
- **PR:** Payload has repo + pr_id when the run was for a PR; link evidence to PR for per-PR views and for PR Intelligence (RFC-016).
- **Bundle:** Optional bundle_id/bundle_version in payload or inferred (e.g. latest bundle for ticket at commit time); used for traceability and “evidence for this bundle” views.

### 2.4 Multi-repo

- **Aggregation key:** Evidence is keyed by (ticket_id, repo, pr_id or commit_sha). For one ticket with PRs in repo A and repo B, there are separate evidence records per repo; aggregation is by ticket_id.
- **Unified view per ticket:** Query “all evidence for ticket X” returns evidence from every repo (list of payloads or summary per repo). PR Intelligence (RFC-016) can show “3/3 repos have evidence” and per-repo pass/fail.
- **“Evidence complete” rules:** Define when a ticket is considered to have sufficient evidence for validation/review. Examples: (1) Every repo in “repos in scope” (from bundle or config) has at least one successful evidence payload. (2) Every PR linked to the ticket has at least one evidence payload with ci_status success. Rules are configurable per tenant or per ticket type. Engine exposes “evidence complete: true/false” and optional “missing: [repos or PRs]” for UI and state machine (RFC-007).
