# RFC-018: Workflow Latency Reduction

**Status:** Draft  
**Summary:** Defines mechanisms to reduce clarification loops and handoff friction: notifications, context handoff format, and role-based (PM/Dev/QA/Reviewer) flows.

---

## 1. Overview

Handoffs between PM, Dev, QA, and Reviewer often introduce delay and repeated context-sharing. This RFC defines mechanisms to reduce that latency: notifications (who gets notified when), context handoff format (what is passed between roles), and role-based flows so that the right information is available at the right step.

**In scope:**

- Notifications: triggers and recipients (e.g. “PR ready for review” → reviewers; “evidence failed” → dev).
- Context handoff format: structured payload (e.g. bundle ref, evidence summary, AC status) that accompanies handoff.
- Role-based flows: PM → Dev → QA → Reviewer; what each role sees and can do.

**Out of scope:**

- Specific notification channels (Slack, email) — abstract “notification”; channels are implementation.
- State machine details (RFC-007) — this RFC uses states/events but does not redefine them.

---

## 2. Mechanisms

### 2.1 Notifications

- **Triggers:** (1) Bundle ready for ticket → notify assignee or dev team. (2) PR opened / evidence submitted → notify reviewers. (3) Evidence validation failed → notify PR author. (4) PR merged → notify QA or release owner. (5) Ticket status change (e.g. In Review) → notify relevant role. Triggers align with state machine (RFC-007).
- **Recipients:** By role (e.g. ticket assignee, PR author, reviewers) or by subscription (e.g. “watch this ticket”). Recipient resolution can use ticket/PR metadata (assignee, reviewers) from adapters (RFC-009, RFC-011).
- **Payload:** Notification includes minimal context: ticket id, PR link (if any), summary line (e.g. “Evidence: 3/3 passed”), and deep link to OR UI or bundle/evidence view. Avoid large attachments; link to source of truth.

### 2.2 Context handoff format

- **Structured handoff:** When ownership or focus moves (e.g. Dev → Reviewer), a “handoff” payload can be generated: ticket_id, bundle_id/version, PR id(s), evidence summary (pass/fail, AC coverage), and optional “next steps” or checklist. Format is machine-readable (e.g. JSON) so that UI or bots can display it consistently.
- **Where it’s used:** (1) In notification body or link (e.g. “View handoff for PROJ-123”). (2) In OR UI when opening a ticket or PR (sidebar “Context for review”). (3) Optional: export for Slack/Teams message or email body.
- **No duplication of full bundle:** Handoff references bundle and evidence by id; full content is fetched on demand via API (RFC-006).

### 2.3 Role-based flows (PM / Dev / QA / Reviewer)

- **PM:** Sees ticket and bundle status; gets notified when bundle is ready or when dev needs clarification. Can update spec/ticket; re-bundle may be triggered (RFC-014). Idea/Spec Assist (RFC-001) supports PM upstream.
- **Dev:** Receives bundle (IDE, RFC-013); gets notified when ticket is assigned or bundle updated. Submits PR; gets notified when evidence fails or review requested. Handoff from PM: “here’s the bundle and AC.”
- **QA:** Gets notified when PR is merged or when “ready for QA” (state or tag). Sees evidence and traceability (RFC-015, RFC-016) for the ticket/PR. Handoff from Dev: “PR merged; evidence summary and test report links.”
- **Reviewer:** Gets notified when PR is ready for review (e.g. evidence passed). Sees PR Intelligence summary and risk flags (RFC-016); can open handoff context. After merge, notification can go to QA or release owner.

Flows are configurable (e.g. skip QA for some ticket types); this RFC defines the pattern, not every variant.
