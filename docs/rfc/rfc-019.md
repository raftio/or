# RFC-019: Outcome Feedback Loop

**Status:** Draft  
**Summary:** Defines how to connect release artifacts to KPI (RFC-005): release identification, metric collection, attribution, and post-release insight format.

---

## 1. Overview

The Outcome Feedback Loop closes the product flow: releases are identified, outcomes (KPIs) are measured, and results are attributed and fed back for product insight. This RFC defines release identification, how metrics are collected and attributed, and the post-release insight format (aligned with RFC-005). It does not define the KPI schema itself (RFC-005) or the Control Plane API (RFC-006).

**In scope:**

- Release identification: how a “release” is defined and stable id assigned.
- Metric collection and attribution: how KPIs are gathered and linked to releases (and optionally to tickets/bundles).
- Post-release insight format: what is stored and exposed (RFC-005).

**Out of scope:**

- Specific data sources (e.g. analytics, APM) — integration patterns only.
- UI for outcome dashboards — data contract only.

---

## 2. Loop

### 2.1 Release identification

- **Definition:** A release is a deployable unit that is shipped (e.g. version tag, deployment id, or build id). For multi-repo: a release may be a set of (repo, tag) or a single “product version” that groups several repo releases.
- **Stable id:** Control Plane assigns or accepts a release id (e.g. `rel_abc123`, or `v1.2.0`). Used in outcome records and attribution. Source: from Git tag webhook (RFC-011), from deployment pipeline callback, or manual “create release” in OR.
- **Link to PRs/tickets:** At release time, link release to the PRs (or commits) that are included, and thus to tickets (and bundles). Enables “what shipped in this release” and “outcome for this ticket/initiative.”

### 2.2 Metric collection and attribution

- **Collection:** Metrics (KPIs) are collected from external systems (analytics, APM, business DB) or from deployment/observability pipelines. OR receives outcome data via API or ingestion: (release_id, kpi_id or metric name, value, window, timestamp). Schema follows RFC-005.
- **Attribution:** Outcome records reference release_id. Optional: attribute to tickets (e.g. “this release included PROJ-123, PROJ-124” so that outcome deltas can be loosely attributed to features). Attribution is best-effort (e.g. based on “tickets in release”); causal attribution is out of scope.
- **Timing:** Collection can be on a schedule (e.g. daily post-release) or event-driven (e.g. “release deployed” → trigger collection after N hours). Stored as outcome records (RFC-005).

### 2.3 Post-release insight format (RFC-005)

- **Outcome record:** release_id, kpi_id, value, aggregation window, timestamp. Optional: comparison to previous release (delta, percent change).
- **Insight view:** For a release: list of outcome records; optional comparison table. For a ticket: “releases that included this ticket” and their outcomes (for product feedback). Format is suitable for API response and UI; details in RFC-005.
