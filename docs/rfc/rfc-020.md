# RFC-020: Auth, Tenancy & Data Isolation

**Status:** Draft  
**Summary:** Defines authentication, authorization, multi-tenancy, and data isolation between teams/orgs.

---

## 1. Overview

The Control Plane serves multiple tenants (e.g. organizations or teams) and must authenticate callers and isolate data per tenant. This RFC defines authentication, the authorization model, and how tenancy and data isolation are enforced. The Control Plane API (RFC-006) relies on this model.

**In scope:**

- Authentication: how users and services prove identity (e.g. API key, OAuth, platform identity).
- Authorization model: roles/capabilities and scoping (e.g. per-tenant, per-project).
- Tenancy and data isolation between teams/orgs.

**Out of scope:**

- Identity provider implementation (e.g. Okta, Auth0) — only the contract (tokens, claims).
- Observability and audit (RFC-021).

---

## 2. Model

### 2.1 Authentication

- **Users:** Login via SSO/OAuth or API key. Orqestra accepts a bearer token or API key and resolves it to an identity (user id, tenant id). Optional: service accounts for CI/IDE with scoped API keys (e.g. “push evidence only”).
- **Tenant context:** Every request is bound to a tenant (org or team). Tenant is derived from token (e.g. claim), API key metadata, or subdomain/header. No cross-tenant access without explicit multi-tenant admin role.
- **Storage:** Credentials and keys are not stored in plain text; verify via identity provider or secure key store.

### 2.2 Authorization model

- **Principle:** All resources (bundles, evidence, tickets, outcomes) belong to a tenant. API operations are allowed only if the identity has the right capability within that tenant.
- **Capabilities:** Examples: read_bundles, write_evidence, manage_integrations, admin_tenant. Roles (e.g. “developer”, “reviewer”, “admin”) map to capability sets. Fine-grained (e.g. “read only project X”) can be added via project/space scoping.
- **Scoping:** Optional project or “space” within a tenant so that some users see only a subset of tickets/bundles (e.g. by Jira project). Default: tenant-wide access for assigned roles.
- **API enforcement:** Every Control Plane API (RFC-006) checks tenant + capability before performing the operation; return 403 when unauthorized.

### 2.3 Tenancy and data isolation

- **Tenant id:** Every entity (bundle, evidence, ticket link, outcome) has a tenant_id. Queries and writes are always filtered by tenant_id. No API can return or modify another tenant’s data.
- **Isolation:** Database and storage use tenant_id in keys or partitions so that logical isolation is enforced. Optional: separate schema or DB per tenant for stronger isolation.
- **Cross-tenant:** Not supported for normal users. Super-admin or support role (if any) is out of scope for this RFC or explicitly restricted (e.g. read-only, audit-logged).
