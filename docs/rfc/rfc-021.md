# RFC-021: Observability & Audit

**Status:** Draft  
**Summary:** Defines logging, metrics, tracing for the Control Plane, and audit trail for intent → execution → evidence → outcome.

---

## 1. Overview

The Control Plane must be observable (logs, metrics, tracing) and key actions must be auditable for compliance and debugging. This RFC defines what is logged and metered, how tracing is applied, and the audit trail for the product flow (intent → execution → evidence → outcome).

**In scope:**

- Logging and metrics: what to log, what to meter, and principles (no secrets, structured logs).
- Tracing: correlation across services and adapters (e.g. trace id from API to evidence sync).
- Audit trail: which actions are recorded and how (immutable, queryable by tenant and time).

**Out of scope:**

- Specific tools (e.g. Datadog, OpenTelemetry) — only the contract and data shape.
- Retention and compliance policies — only that audit records exist and are queryable.

---

## 2. Observability

### 2.1 Logging and metrics

- **Logging:** Structured logs (e.g. JSON) with level, timestamp, tenant_id, and request/correlation id. Log API requests (path, method, status, duration), adapter calls (source, success/failure), and errors with stack or error code. Do not log secrets (tokens, full payloads with credentials).
- **Metrics:** Counters or gauges for: API request rate and latency by endpoint, evidence received count, bundle creation count, validation pass/fail, adapter call success/failure. Optionally: queue depth, cache hit rate. Metrics are tenant-aggregatable or tenant-tagged for multi-tenant dashboards.
- **Alerts:** Out of scope for this RFC; metrics and logs should be sufficient to define alerts (e.g. error rate, latency p99).

### 2.2 Tracing

- **Correlation:** Incoming requests get a trace id (or use incoming header). Propagate trace id to outbound calls (adapters, DB). So that a single “create bundle” request can be traced through context synthesis, ticket fetch, and document fetch.
- **Spans:** Key operations (API handler, adapter call, evidence processing) create spans with name, duration, and optional attributes (e.g. ticket_id, pr_id). Export to tracing backend (e.g. Jaeger, OTLP) for distributed traces.
- **Adapters:** Adapters (RFC-009–013) should accept and forward trace context when calling external APIs so that external latency is visible in the same trace.

### 2.3 Audit trail

- **Scope:** Record actions that change state or access sensitive data: bundle created/updated, evidence received/linked, status updated (ticket), validation result, outcome recorded. Read-only operations (e.g. get bundle) may be logged at debug level but need not be in audit trail.
- **Record format:** Immutable record per action: timestamp, tenant_id, actor (user or service id), action type, resource (e.g. ticket_id, bundle_id), outcome (success/failure), optional old/new value or diff for critical fields. No secrets in audit payload.
- **Storage and query:** Audit records are append-only and queryable by tenant_id and time range (and optionally by actor or resource). Retention and archival policy are operational; this RFC only requires that the trail exists and is queryable for intent → execution → evidence → outcome flow.
